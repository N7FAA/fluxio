## Git 提交约定（本仓库采纳）

以下为**默认习惯**，减少混乱、便于发版时写发版笔记。

### 分支

- **`main`**：可部署分支；与 `package.json` 中当前版本意图一致。
- **功能/修复**：`feat/简短说明`、`fix/简短说明`；合并前本地或 PR 上自测 `npm start` 与 Fluxio 主路径。

### 提交信息格式

采用 **Conventional Commits** 简版，前缀小写英文，冒号后中文或英文均可：

| 前缀 | 用途 |
|------|------|
| `feat` | 新功能或用户可见行为变化 |
| `fix` | 缺陷修复 |
| `style` | 仅样式/UI（Fluxio 的 css/html 视觉调整多用此项） |
| `refactor` | 重构、无行为变化 |
| `chore` | 依赖、脚本、skills、文档元数据等 |
| `docs` | 纯文档 |

示例：`style(fluxio): 调整输出页间距与 Tab 对比度`、`fix(server): 修正任务状态边界`。

### 发版时的 Git 操作（建议）

1. 确认 `package.json` 的 `version` 已为发版号（如 `1.1.0`）。
2. 提交应已包含本次发版相关改动与本文档更新。
3. 打标签（与 `v` 前缀版本号一致）：

```bash
git tag -a v1.1.0 -m "Fluxio UI 更新 v1.1.0"
git push origin v1.1.0
```

4. 服务器上拉取对应 **tag 或 main 上该提交**，再按 `上线操作教程.md` 重启服务。

### 回滚

- **代码回滚**：检出 `v1.0.0` 标签对应提交，或 `git revert` 本次合并提交（择一，团队统一即可）。
- **记录**：回滚后在本文档或下一条发版笔记中**一句话说明原因与日期**即可。
