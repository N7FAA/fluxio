# Fluxio 线上产品 · 可行性分析与技术方案

> 基于 PRD 文档的可行性评估、技术方案及个人开发者实施建议

---

## 一、PRD 核心要点回顾

| 维度 | 内容 |
|------|------|
| **产品名** | Fluxio |
| **Slogan** | 让动效轻松落地 |
| **定位** | 面向线上页面落地的开放转码工具，帮助设计师将图片/动图/视频转为网页可用格式 |
| **三个 Tab** | 图片、动态图像、视频（按**目标输出**分流） |
| **核心能力** | 格式转换、分辨率缩放、质量档位、序列帧支持 |

### 关键约束（PRD 规定）

| Tab | 单文件上限 | 单任务文件数 | 单任务总大小 | 其他 |
|-----|------------|--------------|--------------|------|
| 图片 | 30MB | 30 | 300MB | 单边 ≤8000px |
| 动态图像 | 100MB(GIF/APNG/WEBP) / 500MB(MP4/WEBM/ZIP) / 1GB(MOV) | 10 | 2GB | 视频≤20s，分辨率≤3840×3840，帧数≤600 |
| 视频 | 同上 | 10 | 2GB | 视频≤60s，分辨率≤3840×2160 |

## 二、可行性分析

### 2.1 技术可行性：✅ 可行

| 能力 | 实现方式 | 成熟度 |
|------|----------|--------|
| 图片转换 | Sharp / ImageMagick / FFmpeg | 成熟 |
| 动图转换 | FFmpeg (libwebp, gif, apng) | 成熟 |
| 视频转换 | FFmpeg (libx264, libvpx-vp9) | 成熟 |
| 序列帧 ZIP 解析 | 解压 + 帧排序 + FFmpeg concat | 可实现 |
| 上传/存储 | 对象存储 (S3/OSS) 或本地临时目录 | 成熟 |
| 任务队列 | Bull/BullMQ + Redis / 内存队列 | 成熟 |

**结论**：PRD 中所有功能均有成熟技术栈支撑，无技术黑箱。

### 2.2 成本可行性：⚠️ 需控制

线上转码的核心成本：

| 成本项 | 说明 | 粗略估算 |
|--------|------|----------|
| **计算** | FFmpeg 转码 CPU 密集，单任务可能 10s～数分钟 | 按 vCPU·分钟计费，或按服务器包月 |
| **存储** | 上传临时文件 + 输出文件，需定期清理 | 按 GB·天计费 |
| **带宽** | 上传 + 下载，大文件占比高 | 按流量计费 |
| **基础设施** | 服务器、Redis、对象存储 | 月固定成本 |

**示例**（按个人开发者可接受规模）：
- 单日 100 次转换，平均每任务 50MB 输入 + 20MB 输出
- 存储：约 7GB 峰值（假设 24h 保留）→ 对象存储约 $0.2/月
- 带宽：约 7GB/天 × 30 ≈ 210GB/月 → 约 $20～50/月（视厂商）
- 计算：1 核 2G 服务器可能不够，2 核 4G 约 $20～40/月
- **合计**：约 $50～100/月 可支撑小规模试运行

**风险**：若流量突增或恶意大文件上传，成本会快速上升，需做限流、配额、风控。

### 2.3 合规与隐私：⚠️ 需设计

| 点 | 说明 |
|----|------|
| **数据留存** | 用户上传的设计素材可能含商业机密，PRD 未明确保留时长，需在产品中声明（如 24h 自动删除） |
| **服务条款** | 需有 ToS、隐私政策，说明数据用途与保留策略 |
| **地域** | 若面向国内用户，服务器与存储建议在国内或合规跨境方案 |

### 2.4 竞品与差异化：✅ 有机会

- **CloudConvert、Convertio** 等：功能全但偏通用，Fluxio 聚焦「动效落地」场景，可做更垂直的预设与文案。
- **PRD 的「按目标输出分流」** 与现有 ffmpeg-tool 的 PRODUCT-SPEC 思路一致，体验上有差异化空间。

## 三、技术方案

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│  前端 (SPA)                                                      │
│  · 按 Tab 分流：图片 / 动态图像 / 视频                            │
│  · 上传区、文件列表、输出设置、结果区                             │
│  · 拖拽上传、进度、错误反馈                                       │
└───────────────────────────┬─────────────────────────────────────┘
                            │ HTTPS
┌───────────────────────────▼─────────────────────────────────────┐
│  API 层 (Node.js / Express 或 Fastify)                            │
│  · 上传接口（multipart，写入对象存储或本地 temp）                  │
│  · 任务创建、状态查询、结果下载                                   │
│  · 参数校验、限流、鉴权（若做付费）                               │
└───────────────────────────┬─────────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────────┐
│  任务队列 (Redis + Bull/BullMQ)                                  │
│  · 任务入队、Worker 消费、重试、超时                              │
└───────────────────────────┬─────────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────────┐
│  Worker (Node.js + FFmpeg)                                        │
│  · 按任务类型调用：Sharp/FFmpeg/脚本                              │
│  · 输出写入对象存储或本地，生成下载链接                           │
│  · 任务完成后清理临时文件                                         │
└───────────────────────────┬─────────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────────┐
│  存储                                                             │
│  · 对象存储 (S3/OSS/MinIO)：上传文件、输出文件                    │
│  · 或本地磁盘 + 定期清理（MVP 可简化）                            │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 技术选型建议

| 层级 | 推荐 | 备选 | 说明 |
|------|------|------|------|
| 前端 | React / Vue + Vite | 原生 / Svelte | 组件化、状态管理、上传进度 |
| API | Node.js + Express | Fastify | 与现有 ffmpeg-tool 技术栈一致 |
| 队列 | Bull + Redis | 内存队列 | 生产环境建议 Redis，MVP 可先用内存 |
| 转码 | FFmpeg + Sharp | ImageMagick | Sharp 处理静态图更轻量 |
| 存储 | 本地 temp + 定时清理 | S3/OSS | MVP 用本地，上量后迁对象存储 |
| 部署 | 单机 Docker | 云函数 + 队列 | 单机足够支撑早期流量 |

### 3.3 核心接口设计

```
POST   /api/upload          # 分片或整体上传，返回 fileId
POST   /api/tasks           # 创建任务 { tab, fileIds[], outputConfig }
GET    /api/tasks/:id       # 任务状态、进度、结果列表
GET    /api/tasks/:id/files/:fileId/download  # 下载单个结果
GET    /api/tasks/:id/download-all            # 打包下载（ZIP）
DELETE /api/tasks/:id       # 取消/清理任务
```

### 3.4 转码逻辑与 FFmpeg 映射

| Tab | 输入→输出 | FFmpeg/工具 要点 |
|-----|-----------|------------------|
| 图片 | PNG/JPG/WEBP → PNG/JPG/WEBP | Sharp 更合适；FFmpeg 也可 |
| 动态图像 | 动图/视频/序列帧 → GIF/APNG/WEBP | libwebp、gif、apng；视频用 -vf fps、-t 截时长 |
| 视频 | 动图/视频/序列帧 → MP4/WEBM | libx264、libvpx-vp9；-crf、-preset 对应质量档 |

**质量档位映射**（5 档，见 FLUXIO-INTERACTION-FINAL 十、质量滑块）：
| 档位 | 位置 | 透出文案 | CRF/参数 |
|------|------|----------|----------|
| 0% | 体积优先 | 体积优先 | 高 CRF、低码率 |
| 25% | 偏小 | 不显示 | 偏高 CRF |
| 50% | 推荐 | 推荐 | 中间值 |
| 75% | 偏清晰 | 不显示 | 偏低 CRF |
| 100% | 清晰优先 | 清晰优先 | 低 CRF、高码率、保持分辨率 |

**序列帧 ZIP**：
1. 解压到临时目录
2. 命名规范：`前缀_编号`，如 `demo_0001.png`。同一 ZIP 内前缀需完全一致，编号为 4–6 位纯数字、位数一致，从 0001 开始连续递增
3. 校验：命名、尺寸一致、帧数≤600
4. 用 `ffmpeg -framerate X -i 前缀_%04d.png -c:v ...` 合成

### 3.5 风险与应对

| 风险 | 应对 |
|------|------|
| 大文件占满磁盘 | 单任务大小限制、定时清理、对象存储生命周期 |
| 恶意刷量 | 限流、IP 限制、简单验证码 |
| FFmpeg 超时/崩溃 | 任务超时、Worker 重启、失败重试 |
| 并发过高 | 队列并发数限制、排队提示 |

## 四、个人开发者实施建议

### 4.1 若是我，会怎么做

#### 阶段 0：验证需求（1～2 周）
- 不做完整产品，先做一个**极简落地页 + 等待名单**，或发到设计/前端社群收集反馈
- 明确：是否有人愿意用、愿意为「省时间」付费
- 若有现有 ffmpeg-tool 用户，可定向邀请试用线上版

#### 阶段 1：MVP（4～6 周）
- **范围收缩**：
  - 只做「图片」+「动态图像」两个 Tab，视频 Tab 延后
  - 序列帧 ZIP 延后，先支持单文件动图/视频
  - 单任务文件数：图片 10、动图 5
- **技术从简**：
  - 单机部署，无 Redis，用内存队列
  - 上传存本地 `temp/`，24 小时后定时任务清理
  - 不做用户系统，不做付费
- **目标**：跑通「上传 → 转换 → 下载」全流程，验证 PRD 主流程是否成立

#### 阶段 2：可用的免费版（2～3 个月）
- 补齐「视频」Tab、序列帧、批量
- 引入 Redis + Bull，支持多 Worker 或单 Worker 多进程
- 增加限流、错误提示、结果页压缩比展示（PRD 16.3）
- 明确数据保留策略（如 24h 删除），写进隐私说明
- 可选：简单统计（日活、任务数）用于迭代

#### 阶段 3：商业化探索（按需）
- 若流量稳定，考虑：对象存储、CDN、更大算力
- 付费点：更高配额、API、去水印、优先队列等
- 或保持免费，用引流到其他产品/服务

### 4.2 优先级建议

| 优先级 | 功能 | 理由 |
|--------|------|------|
| P0 | 图片 Tab 全流程 | 实现简单，验证产品形态 |
| P0 | 动态图像 Tab（单文件，不含序列帧） | 核心场景，GIF/WEBP 转视频需求大 |
| P1 | 视频 Tab | 与动图共享输入，扩展价值高 |
| P1 | 序列帧 ZIP | 需求存在但实现复杂，可后置 |
| P1 | 批量 + 队列体验 | 提升效率感 |
| P2 | 用户系统、付费 | 有稳定流量后再做 |

### 4.3 成本控制原则

- **免费额度**：如每日 10 次转换，超出需排队或引导付费
- **文件大小**：严格按 PRD 限制，避免单任务过大
- **保留时长**：24h 自动删除，降低存储成本
- **监控**：磁盘、内存、任务堆积，超阈值告警

### 4.4 与现有 ffmpeg-tool 的关系

- **ffmpeg-tool**：本地工具，适合「本地文件、无隐私顾虑、参数可调」场景
- **Fluxio 线上版**：适合「无安装、跨设备、快速出结果」场景
- 可共享：FFmpeg 参数逻辑、预设配置、部分 UI 组件
- 建议：先基于 ffmpeg-tool 的 server 逻辑抽一层「转码服务」，线上版只做 API + 队列 + 存储适配

## 五、总结

| 维度 | 结论 |
|------|------|
| **技术可行性** | ✅ 高，无技术障碍 |
| **成本** | ⚠️ 需限流与配额控制，MVP 可控制在较低成本 |
| **合规** | ⚠️ 需明确数据保留与隐私政策 |
| **差异化** | ✅ 聚焦动效落地，有差异化空间 |
| **个人开发者路径** | 先 MVP 验证 → 免费版打磨 → 再考虑商业化 |

**建议**：从「图片 + 动态图像」双 Tab 的 MVP 起步，单机部署、本地存储、无用户系统，用 4～6 周跑通主流程并小范围试运行，再根据反馈决定是否投入序列帧、视频 Tab 和基础设施升级。
